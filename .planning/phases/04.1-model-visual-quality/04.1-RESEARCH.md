## Summary

This phase replaces all placeholder geometric player-visible models (third-person soldier, first-person arms, rifle/pistol/knife) with realistic, commercially-licensed assets. The Mixamo Vanguard character (or equivalent Mixamo soldier) is the locked choice for the third-person model. Weapons and FP arms need separate sourcing. The dominant technical risk is rig migration: the existing procedural animation in `src/player-model.js` uses named Euler rotations designed for a custom 19-bone armature. Mixamo characters use a 65-bone skeleton with different rest-pose bone orientations. Applying the existing rotation values directly to Mixamo bones will produce wrong poses.

The recommended migration approach is: (A) import Mixamo FBX into Blender, strip finger bones, rename bones to match the current custom names, bake the neutral (T/A-pose) as rest pose, then export GLB — so the existing procedural code works without modification. This is lower risk than rewriting all animation math. Option B (adapt animation code to Mixamo local axes) is a deeper but more maintainable path if the project outgrows the current placeholder animations. Option C (blend baked Mixamo clips with procedural overlays) is the most powerful but requires the most new code.

For weapons: Fab.com paid packs (Standard License - Professional tier) or Sketchfab CC0 packs are the clearest commercial-safe routes. CGTrader royalty-free purchases also qualify.

**Primary recommendation:** Download Mixamo character with "No Fingers (25)" skeleton LOD, import to Blender, rename bones to current custom names (Root/Spine/Chest/Neck/Head/Shoulder.L/.R/UpperArm.L/.R/ForeArm.L/.R/Hand.L/.R/Thigh.L/.R/Shin.L/.R/Foot.L/.R), bake T-pose as rest pose, export GLB. This reuses the entire existing `player-model.js` procedural system unchanged. Source weapons from a Fab.com paid pack with Standard License (Professional tier covers commercial games of any revenue level).

## Architecture Patterns

### Recommended Asset Pipeline

```
1. mixamo.com → download Vanguard (or similar soldier) → FBX, T-pose, "No Fingers (25)" skeleton
2. Blender 5.0:
   a. Import FBX (Manual Orientation, Automatic Bone Orientation)
   b. Rename bones: strip "mixamorig:" prefix, Left→.L/Right→.R, map to custom names (see table below)
   c. Apply current T-pose as rest pose (Pose Mode → Apply → Apply Pose as Rest Pose)
   d. Export GLB (export_skins=True, export_yup=True, no animations)
3. gltf-transform CLI: compress textures to 1024x1024 WebP, meshopt geometry compression
4. Drop into visual/ folder replacing mannequin_neutral.glb
```

### Blender Python Script Pattern for Bone Rename

```python
import bpy

# Rename mapping: Mixamo (no prefix) → custom
RENAME_MAP = {
    'Hips': 'Root',
    'Spine': 'Spine',
    'Spine1': 'Chest',
    'Spine2': 'Chest2',  # extra bone; procedural code ignores unknown names
    'Neck': 'Neck',
    'Head': 'Head',
    'LeftShoulder': 'Shoulder.L',
    'LeftArm': 'UpperArm.L',
    'LeftForeArm': 'ForeArm.L',
    'LeftHand': 'Hand.L',
    'RightShoulder': 'Shoulder.R',
    'RightArm': 'UpperArm.R',
    'RightForeArm': 'ForeArm.R',
    'RightHand': 'Hand.R',
    'LeftUpLeg': 'Thigh.L',
    'LeftLeg': 'Shin.L',
    'LeftFoot': 'Foot.L',
    'RightUpLeg': 'Thigh.R',
    'RightLeg': 'Shin.R',
    'RightFoot': 'Foot.R',
}

armature = bpy.data.armatures['Armature']  # adjust name as needed
for bone in armature.bones:
    # Strip mixamorig: prefix first
    clean_name = bone.name.split(':')[-1]
    if clean_name in RENAME_MAP:
        bone.name = RENAME_MAP[clean_name]
    else:
        bone.name = clean_name  # strip prefix, keep name
```

### The Mixamo Rest-Pose Problem and Why "Apply Rest Pose" Solves It

**The problem:** Mixamo characters download in an A-pose (arms angled 45° down from horizontal) or T-pose. Even after renaming bones, the Mixamo `UpperArm.L` bone's local X axis points roughly along the bone's long axis (toward the elbow), whereas the custom mannequin's `UpperArm.L` has its local X axis pointing differently because of how the Blender armature was originally built. The existing code does `bones['UpperArm.L'].rotation.x = Math.sin(t)`, which means "rotate around the bone's local X axis by that amount." On the custom mannequin this swings the arm forward/backward (correct). On the Mixamo bone with different local axes, `rotation.x` may swing sideways.

**The solution:** In Blender, pose the Mixamo armature so the arms are in the same neutral position as the original mannequin (arms hanging straight down, matching the original mannequin's rest pose), then apply "Pose as Rest Pose" (Pose Mode → Pose menu → Apply → Apply Pose as Rest Pose). This bakes the current pose into the rest pose, resetting all bone rotations to zero. After this, the local axes of each renamed bone will be in an orientation where the procedural Euler rotations produce roughly the same visual result as on the original mannequin. **This approach allows reusing the entire existing `player-model.js` without code changes.**

**Caveat (MEDIUM confidence):** The local X/Y/Z axes of Blender-created bones vs Mixamo-imported-then-exported bones may still differ subtly even after apply-rest-pose. The safest way to verify is: after loading the new GLB in-game, set `debugMode = true` to show SkeletonHelper, then run the existing run animation and inspect whether legs swing forward/backward correctly. If an axis is wrong (e.g., arms swing sideways instead of forward), the fix is a one-line code change: swap `rotation.x` to `rotation.z` or `-rotation.x` for that bone.

### Alternative Approach B: Adapt Animation Code to Mixamo Axes (MEDIUM confidence)

If approach A produces subtle incorrectness or the project will add many more animation states, a cleaner path is to keep the Mixamo rest pose as-is and update the animation code:

1. Load the renamed Mixamo GLB in-game.
2. Enable `debugMode`, attach `AxesHelper` to each animated bone to see local axis directions.
3. For each bone in `_animateRun`, `_animateCrouch`, etc., empirically determine the correct rotation axis (x/y/z) and sign.
4. Update `player-model.js` with corrected axes.

This is more work up-front but results in correct animations that match the Mixamo character's natural proportions. Estimated effort: 2–4 hours of in-game tuning.

### Alternative Approach C: Baked Clip Blending (LOW–MEDIUM confidence)

Use Mixamo's Idle, Walk, and Run animations as the base movement layer, then procedurally overlay shoot/reload/knife on top using Three.js `AnimationMixer` with masking:

```javascript
// Base movement: baked Mixamo Run clip on lower body
const runAction = mixer.clipAction(runClip);
runAction.play();

// Override upper body with procedural shooting pose each frame
// (set bone.quaternion after mixer.update() to override lower priority)
mixer.update(dt);
bones['UpperArm.R'].rotation.x -= recoil * 0.15; // additive after mixer
```

**Tradeoffs:** The baked clips will look more natural for locomotion (Mixamo animators vs. sine waves), but blending procedural overlays on top of baked clips is complex — you need to understand AnimationMixer's bone mask system (Three.js does not have a simple additive layer system without custom setup). This is the highest quality path but also the highest risk/effort. **Not recommended for Phase 4.1.** Defer to a future polish phase.

### Team Color Approach

**Recommended: emissive accent + material.color tint**

The Mixamo Vanguard has a baked PBR albedo texture (tan camo pattern). Simply setting `material.color` (as the current code does) will multiply the albedo texture by that color, darkening/tinting it. For RED team this gives a reddish-tan camo; for BLUE a bluish-tan camo. This is readable but subtle.

For clear team identification, use `material.emissive` on secondary material slots (gloves, vest, helmet strap) to add glowing color accents. The Vanguard mesh may have multiple material slots; identify which slots cover vest/belt/helmet accessories and set those to emit team color.

**If the Vanguard has only a single material:** Use `onBeforeCompile` to inject a vertex-colored overlay for armband/kneepads — mark those vertices with vertex color in Blender, then the shader multiplies or adds team color in those areas.

**Fallback (simplest):** Keep the current approach — `material.color.setHex(teamColor)` tints the whole texture. Increase the team color saturation so it reads clearly (0xff0000 red, 0x0000ff blue) even if the camo texture desaturates it slightly. This requires zero new code.

**Recommended implementation order:**
1. Start with `material.color` tint (zero code change — current code already does this).
2. If playtesting shows colors are too ambiguous, add emissive accent via `material.emissive.setHex(teamColor)` with `emissiveIntensity = 0.3` on the same material.
3. If a single emissive is still unclear, add a small team-colored object (armband/insignia plane) attached to the Chest bone in Blender.

## Common Pitfalls

### Pitfall 1: Mixamo Bone Local Axes Cause Wrong Animation Directions
**What goes wrong:** After loading the Mixamo GLB, `bones['UpperArm.L'].rotation.x = 0.4` swings the arm sideways instead of forward.
**Why it happens:** Mixamo arm bones' local X axis does not point in the same direction as the custom mannequin's arm bones. In Mixamo, the arm bone local X axis typically points upward (along the character's global Y) when the arm is in the rest A-pose.
**How to avoid:** Use the "Apply Pose as Rest Pose" approach in Blender described above. After baking, verify in-game with SkeletonHelper + debugMode. If a bone still animates wrong, check which axis gives the correct direction empirically (swap .x to .z etc.).
**Warning signs:** Leg swings sideways during run, arm swings in wrong plane.

### Pitfall 2: Spine2 Extra Bone Breaks Skeleton Clone
**What goes wrong:** The Mixamo 25-bone skeleton has 3 spine bones but the ragdoll/hitbox systems reference by name. Having an extra `Chest2` bone (or Spine2) causes no crash but the hitbox for torso maps to the wrong position if the bone lookup returns the wrong one.
**Why it happens:** The existing code does `bones[name] = bone` in a flat map, so if there are two bones with similar names, the last one wins.
**How to avoid:** In Blender, delete Spine2 from the Mixamo rig (merge its children to Chest/Spine1) before export. The 25-bone skeleton only needs to be trimmed to 19–20 bones matching the custom rig.
**Warning signs:** Chest ragdoll body part appears at wrong position, hitbox for torso is misplaced.

### Pitfall 3: Mixamo FBX Import at Wrong Scale (0.01 scale)
**What goes wrong:** Mixamo FBX files export at centimeter scale (0.01 in Blender). The soldier appears 1.8cm tall instead of 1.8m.
**Why it happens:** Mixamo uses centimeters for FBX export; Blender interprets this without auto-scaling by default.
**How to avoid:** When importing FBX in Blender, ensure Scale is set to 0.01 (auto-detected) OR use File > Import > FBX with "Manual Orientation" and check that the unit scale is correct. After import, run `bpy.ops.object.transform_apply(scale=True)` to bake the scale. Verify height is 1.8m (same as player capsule) before export.
**Warning signs:** Character is tiny or huge; ragdoll capsule sizes are completely wrong.

### Pitfall 4: Weapon Axis Alignment After Model Swap
**What goes wrong:** New realistic rifle model points up or sideways when attached to Hand.R bone.
**Why it happens:** Weapon models from asset stores are authored with varying axis conventions. The existing code applies `rifle.rotation.y = -Math.PI/2` assuming the model was built along the +X axis (as the original Blender weapons were).
**How to avoid:** After loading the new weapon GLB, check the forward direction. Use a SkeletonHelper or add a temporary AxesHelper to the weapon. Adjust the hardcoded rotation offset for each weapon model.
**Warning signs:** Weapon points at the ceiling when held; muzzle flash appears behind the camera.

### Pitfall 5: PBR Textures Are Too Large for Web Delivery
**What goes wrong:** Realistic weapon or soldier GLB loads slowly or causes memory pressure (textures can be 4K = 64MB+ uncompressed on GPU).
**Why it happens:** Asset store packs often ship 4K textures for high-fidelity renders; browser games need much smaller.
**How to avoid:** After purchasing/downloading assets, run gltf-transform to resize to 1024x1024 WebP. For the soldier character, 2048x2048 albedo is acceptable but 4K is not needed at typical in-game distance. Weapon FP textures can be 1024x1024.
**Warning signs:** First load takes >3 seconds on good connection; GPU memory warnings in browser.

### Pitfall 6: Prototype Scaffolding Left in Production
**What goes wrong:** `soldier_proto.glb` is loaded on every game start via `_loadSoldierProto()` in `visual/main.js`.
**Why it happens:** Left in from prototype work (confirmed in `main.js` lines 370–405 and the `soldierMixer.update(delta)` at line 1411).
**How to avoid:** The plan must include explicit tasks to remove `_loadSoldierProto()`, the `this.soldierMixer` update call, and the `soldier_proto.glb` file from `visual/`.
**Warning signs:** An extra unanimated or idle-animated figure appears next to mannequins at game start.

## Performance Budgets

### Character Model
| Metric | Budget | Rationale |
|--------|--------|-----------|
| Triangle count (soldier) | 8,000–15,000 tris | Soldier.glb (Three.js example) is ~7K tris and looks good; 15K is ceiling |
| Texture size (albedo) | 1024×1024 or 2048×2048 | 2048 acceptable for hero character; resize with gltf-transform |
| Bone count | 19–25 (no fingers) | Download Mixamo with "No Fingers (25)" option; strip to 19–20 |
| Draw calls per character | 1–3 | Combine body + equipment into 1–2 material groups in Blender |
| Instances | 4–8 (1v1/2v2/4v4) | SkeletonUtils.clone() handles multiple instances |

### Weapon Models
| Metric | Budget | Notes |
|--------|--------|-------|
| Triangle count | 3,000–8,000 tris per weapon | FP weapon can be higher poly (closer to camera); TP can be lower |
| Texture size | 512×512 or 1024×1024 | Weapons are small on screen; 512 often sufficient |
| Materials | 1 per weapon | Use material atlas to avoid extra draw calls |

### Compression
- **Geometry:** meshopt (EXT_meshopt_compression) via gltf-transform — faster decompression than Draco, no separate decoder needed
- **Textures:** WebP or KTX2 via gltf-transform — 60–80% size reduction vs PNG/JPEG
- **Draco:** Use only if targeting maximum compatibility with old Three.js setups; for this project meshopt is preferred

```bash
# One-time post-process command per asset
npx @gltf-transform/cli optimize soldier.glb soldier_opt.glb \
  --texture-compress webp \
  --texture-size 1024 \
  --meshopt
```

Note: Draco/meshopt compression reduces file transfer size and parse time, NOT GPU runtime performance. To improve framerate: reduce triangle count and draw calls, not just compress.

### Expected Performance (4 players at 60fps)
- 4x SkinnedMesh (15K tris each) = 60K tris total — well within browser WebGL budget
- Bone update cost at 19 bones × 4 instances = 76 bone transforms per frame — negligible
- Main bottleneck remains draw calls from the map (153 mesh objects), not characters

## Ragdoll Remap Implications

The existing ragdoll in `src/ragdoll.js` reads bone world positions by name (lines 119–175). After the rename approach, the bone names will be identical to current names, so no ragdoll code changes are needed.

**One change required:** The capsule sizes in `spawnRagdoll` (e.g., head radius 0.12, chest halfHeight 0.15) were calibrated for the custom mannequin's geometric-primitive body parts. The Mixamo soldier has different proportions. The capsule sizes will need re-calibration once the new character is loaded — this is a minor tuning task, not a refactor.

**Finger bones:** Using the "No Fingers (25)" download means no finger bones in the skeleton. The ragdoll doesn't reference finger bones, so this is a non-issue.

**If 65-bone Mixamo (with fingers) is used:** The bones map will have entries like `mixamorig:LeftHandIndex1` etc. after renaming. These are simply extra entries in the `bones` object that the ragdoll code never references — harmless. The 12 physics bodies in the ragdoll map to the 12 named bones it uses, unchanged.

## Rig Migration Decision: Final Recommendation

| Option | Effort | Risk | Code changes | Quality |
|--------|--------|------|--------------|---------|
| A: Rename + bake rest pose in Blender | 3–6 hours Blender work | Low — procedural code unchanged | None to player-model.js | Good — Mixamo body, existing animations |
| B: Rename + update animation axis mapping | 6–12 hours | Medium — requires in-game axis testing | Minor per-bone fixes to player-model.js | Better — animations match character proportions |
| C: Baked Mixamo clips + procedural overlay | 12–24 hours | High — AnimationMixer blending complexity | Significant new code | Best locomotion quality |

**Choose Option A for Phase 4.1.** The goal is asset replacement with minimum engine risk. Option B can be layered on afterward if tuning reveals issues. Option C is a separate future phase.

**Option A execution steps:**
1. Download Mixamo character FBX (T-pose, No Fingers 25)
2. Blender: import FBX, verify 1.8m height, rename bones (Python script above)
3. Blender: pose to match custom mannequin neutral (arms hanging down, matching mannequin_neutral.glb rest pose exactly)
4. Blender: Apply Pose as Rest Pose
5. Blender: export GLB with export_skins=True, export_yup=True, no animations
6. gltf-transform: compress textures to 1024 WebP
7. In-game: enable debugMode, verify run/strafe/jump/shoot animations look correct
8. Minor axis sign/swap fixes to player-model.js if needed (MEDIUM confidence step 7 will need minimal tuning)
