    Implement the core CS:S movement algorithm based on the Source SDK GameMovement.cpp (see 01-RESEARCH.md Pattern 3).

    **Player.ts:**
    - Player class holding all state:
      - position: THREE.Vector3 (world position, feet)
      - velocity: THREE.Vector3 (units per second)
      - yaw: number (horizontal look angle in radians, driven by mouse X)
      - pitch: number (vertical look angle in radians, driven by mouse Y, clamped to +/-89 degrees)
      - isGrounded: boolean
      - isCrouching: boolean
      - eyeHeight: number (1.7 standing, ~0.94 crouching -- 45% reduction from 1.7)
      - height: number (1.8 standing, ~1.0 crouching)
      - radius: number (0.4 -- capsule radius for collision)
    - Constants (export as named constants or static):
      - STAND_HEIGHT = 1.8
      - CROUCH_HEIGHT = 1.0
      - STAND_EYE_HEIGHT = 1.7
      - CROUCH_EYE_HEIGHT = 0.94
      - PLAYER_RADIUS = 0.4
    - clone() method for state interpolation in game loop

    **Movement.ts:**
    - Movement class implementing the Source SDK movement model
    - CS:S constants (exact values, configurable):
      - SV_FRICTION = 4.0
      - SV_ACCELERATE = 5.0
      - SV_AIRACCELERATE = 10.0
      - SV_STOPSPEED = 100.0
      - SV_MAXSPEED = 250.0 (knife speed -- only weapon in Phase 1)
      - SV_GRAVITY = 800.0
      - SV_JUMPSPEED = 301.993377 (sqrt(2 * 800 * 57.0) -- CS:S jump height)
      - AIR_SPEED_CAP = 30.0
    - Core methods (MUST follow Source SDK order and formulas exactly):

      **friction(player, dt):**
      - Only applies when grounded
      - speed = horizontal velocity magnitude (XZ plane only, ignore Y)
      - if speed < 0.1, set horizontal velocity to zero and return
      - control = max(speed, SV_STOPSPEED)
      - drop = control * SV_FRICTION * dt
      - newSpeed = max(speed - drop, 0)
      - Scale horizontal velocity by (newSpeed / speed)
      - CRITICAL: Apply friction BEFORE acceleration each tick (order matters for counter-strafing)

      **accelerate(player, wishDir, wishSpeed, accel, dt):**
      - currentSpeed = velocity.dot(wishDir) (projection of velocity onto wish direction, XZ plane)
      - addSpeed = wishSpeed - currentSpeed
      - if addSpeed <= 0, return (already at or above wished speed in this direction)
      - accelSpeed = accel * dt * wishSpeed
      - if accelSpeed > addSpeed, accelSpeed = addSpeed
      - velocity += wishDir * accelSpeed
      - This is the standard ground acceleration

      **airAccelerate(player, wishDir, wishSpeed, accel, dt):**
      - cappedWishSpeed = min(wishSpeed, AIR_SPEED_CAP)  -- THIS IS THE KEY TO BHOP
      - currentSpeed = velocity.dot(wishDir)
      - addSpeed = cappedWishSpeed - currentSpeed
      - if addSpeed <= 0, return
      - accelSpeed = accel * wishSpeed * dt
      - if accelSpeed > addSpeed, accelSpeed = addSpeed
      - velocity += wishDir * accelSpeed
      - The air speed cap (30.0) combined with projection-based limiting is what allows perpendicular velocity stacking (bhop speed gain)

      **getWishDirection(input, player):**
      - Build wish direction from input snapshot and player yaw:
        - forward/back input maps to camera forward direction (XZ plane, normalized)
        - left/right input maps to camera right direction (XZ plane, normalized)
        - Sum the components, normalize the result
        - Return the normalized wish direction (Y component always 0)
      - If no movement keys pressed, return zero vector

      **update(player, input, dt):**
      - The main per-tick update function. Order is critical:
        1. Update yaw and pitch from input.mouseDelta (yaw -= dx * sensitivity, pitch = clamp(pitch - dy * sensitivity, -89deg, 89deg))
        2. Calculate wishDir and wishSpeed from input
        3. If grounded:
           a. Apply friction FIRST: friction(player, dt)
           b. Then accelerate: accelerate(player, wishDir, wishSpeed, SV_ACCELERATE, dt)
           c. Handle jump: if input.jump AND grounded, set velocity.y = SV_JUMPSPEED, set isGrounded = false
        4. If airborne:
           a. airAccelerate(player, wishDir, wishSpeed, SV_AIRACCELERATE, dt)
        5. Apply gravity: velocity.y -= SV_GRAVITY * dt
        6. Handle crouch: if input.crouch, set isCrouching = true (adjust height/eyeHeight), else set false
           - When crouching in air, pull feet up (position.y stays, height shrinks from bottom)
           - When uncrouching, check headroom before allowing (prevent standing inside geometry)
        7. Integrate position: position += velocity * dt
        8. (Collision is handled externally after this step, see Task 2)

    - Mouse sensitivity: default 0.002 radians per pixel of movementX/Y. Store as a constant that will be made configurable in Plan 03.

    CRITICAL IMPLEMENTATION NOTES:
    - All velocity calculations use XZ horizontal plane for speed/friction/acceleration. Y (vertical) is only affected by gravity and jump.
    - Do NOT clamp total velocity magnitude. The projection-based acceleration limiting in accelerate/airAccelerate naturally handles speed caps. Direct velocity clamping breaks bunny hopping.
    - Friction BEFORE acceleration each tick. This is what makes counter-strafing work (friction brings you close to zero, then acceleration in opposite direction finishes the stop).
    - Use Three.js Vector3 for all math. Create temporary vectors carefully to avoid garbage collection pressure (reuse where possible).

    Build the collision system and wire physics into the existing game loop.

    **Collision.ts:**
    - Collision class that resolves player AABB against world Box3 array
    - Player collision shape: axis-aligned box (width = 2*PLAYER_RADIUS, height = player.height, depth = 2*PLAYER_RADIUS) centered on player.position (feet at position.y, top at position.y + height)
    - Constructor takes the collision boxes array from TestArena.getCollisionBoxes()

    - **resolve(player: Player, previousPosition: Vector3):**
      Main collision resolution method called AFTER movement integration each tick.

      Strategy: swept AABB with iterative resolution
      1. Build player AABB at new position (after movement integration)
      2. For each world box, test intersection with player AABB
      3. If intersection found:
         - Determine penetration on each axis (X, Y, Z)
         - Push player out along the axis of minimum penetration (this creates sliding behavior)
         - Zero out velocity component on that axis
      4. Repeat intersection tests (up to 3 iterations) to handle corner cases where resolving one collision creates another

    - **Ground detection:**
      - After collision resolution, cast a short ray downward from player position (length = 0.05 units)
      - If ray hits a surface within 0.05 units below player feet: player.isGrounded = true
      - Otherwise: player.isGrounded = false
      - Alternative approach: check if player's Y velocity was zeroed by collision AND player is resting on a surface (velocity.y was negative or zero and got pushed up)
      - The grounded state is critical for deciding friction vs air physics next tick

    - **Ramp handling:**
      - When player moves along a ramp surface (collision normal is angled, not purely vertical), project velocity onto the ramp surface plane so the player glides up/down ramps smoothly
      - Detect ramp by checking if the collision normal Y component is between 0.5 and 0.97 (not flat ground, not a wall)
      - On ramps: still consider the player "grounded" (ground friction and acceleration apply)

    - **Crouch collision:**
      - When player wants to uncrouch, check if there's geometry above that would block standing up
      - If blocked: keep crouching regardless of input
      - Test by creating a test AABB at standing height and checking for intersection

    **GameLoop.ts modifications:**
    - Update the physics step callback to integrate Movement and Collision:
      1. Poll input from InputManager
      2. Call movement.update(player, input, dt) -- updates velocity and integrates position
      3. Call collision.resolve(player, previousPosition) -- resolves penetrations and sets grounded state
      4. Update camera position to match player position + eye height
      5. Update camera rotation to match player yaw/pitch

    - The render callback interpolates between previous and current player position for smooth camera movement between physics ticks. This means:
      - Store previousPlayer state before physics step
      - In render: camera.position = lerp(previousPlayer.position, currentPlayer.position, alpha) + (0, eyeHeight, 0)

    **main.ts modifications:**
    - Create Player instance at spawn position (0, 0.1, 0) -- slightly above ground to let gravity settle
    - Create Movement instance
    - Create Collision instance with TestArena collision boxes
    - Wire into GameLoop physics step
    - Camera position now tracks player (not static at 0, 1.7, 0)

    CRITICAL:
    - Camera rotation (yaw/pitch from mouse) should feel identical to before physics integration. The mouse should NOT feel different after adding movement.
    - Do not apply any position smoothing to camera beyond the fixed-timestep interpolation. No easing, no lerping camera to target. Camera position = interpolated player position + eye height.
    - Player starts slightly above ground and settles via gravity on first tick (avoids spawning inside floor).
