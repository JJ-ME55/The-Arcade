Key architecture decisions:
- Raw Three.js (not R3F) loaded via CDN ESM imports
- Fixed-timestep 64Hz physics simulation with accumulator pattern
- CS:S movement physics (ground friction 4.0, ground accel 5.0, air accel 10.0, gravity -980, jump 270)
- First-person camera with pointer lock
- Player capsule: radius ~0.35 meters, height ~1.8 meters (72 HU in Source scale)
- The GLB was exported from a Blender file using Z-up; GLTF exporter converts to Y-up automatically

Scale consideration:
- The Blender map uses Blender units where the arena spans roughly -20 to +20 on X and -25 to +30 on Y
- In Three.js (after GLTF Y-up conversion), these Blender units become meters
- The Phase 1 movement engine uses Hammer Units (HU) where player is 72 tall, speed is 215 HU/s
- CRITICAL: The map is in ~meter scale (40 units wide), but Phase 1 physics uses HU scale (player 72 units tall)
- You must EITHER scale the map up to HU scale OR adjust physics constants to meter scale
- Recommended: Adjust physics constants to meter scale since the map is already in meters
  - Player height: ~1.8m (not 72 HU)
  - Gravity: -9.8 m/s^2 (not -980 HU/s^2, but note CS:S uses ~800 units/s^2 gravity which is ~20 m/s^2 for faster feel)
  - Jump impulse: ~5.5 m/s (not 270 HU/s; 270 HU/s / 48 = 5.625 m/s where 48 is rough HU-to-meter)
  - Max speed: ~4.5 m/s (215 HU/s / 48 ≈ 4.5 m/s)
  - Ground friction/accel: Keep the same dimensionless multipliers (4.0 friction, 5.0 accel, 10.0 air accel)
  - Actually, the exact HU-to-meter conversion factor is: 1 HU = 0.01905m (1 inch), so 72 HU = 1.3716m
  - But the Blender map wasn't built to HU scale -- it was custom-built in Blender meter units
  - So scale the player/physics to match the map, not vice versa
  - Use the map geometry to determine correct scale: if doorways are ~2 Blender units wide, and a player should fit through, capsule radius should be ~0.35

Three.js Octree + Capsule usage (from research):
- Import Octree from 'three/examples/jsm/math/Octree.js'
- Import Capsule from 'three/examples/jsm/math/Capsule.js'
- worldOctree.fromGraphNode(gltfScene) to build collision from loaded map
- playerCollider = new Capsule(start, end, radius) for player shape
- result = worldOctree.capsuleIntersect(playerCollider) for collision query
- result.normal = surface normal, result.depth = penetration depth
- Collision response: push player out by normal * depth, remove velocity into surface

Spawn points:
- Extract from GLTF scene by checking object.name for "spawn_red" / "spawn_blue"
- OR check object.userData.spawnTeam for "red" / "blue"
- Fallback positions if not found: red at (0, 0, 23) and blue at (0, 0, -28) -- note Y/Z swap from Blender

Lighting (MAP-06):

- DirectionalLight(0xffffff, 0.8) from above-diagonal, castShadow=false
- 1-2 additional fill DirectionalLights (intensity 0.3) from other angles to eliminate dark corners
- Scene background: overcast blue-grey (0x8899aa or similar)

F) **Collision resolution in _simulateTick:**
After integrating velocity into position:
1. Update capsule position to match player position:
   ```js
   this.playerCollider.start.set(pos.x, pos.y + 0.35, pos.z);
   this.playerCollider.end.set(pos.y + standingHeight - 0.35, pos.z);
   // Actually: start.set(x, y + radius, z), end.set(x, y + height - radius, z)
   ```
2. Query collision: `const result = this.worldOctree.capsuleIntersect(this.playerCollider);`
3. If result:
   - Push out: `pos.addScaledVector(result.normal, result.depth);`
   - Remove velocity into surface:
     ```js
     const velIntoSurface = vel.dot(result.normal);
     if (velIntoSurface < 0) {
       vel.addScaledVector(result.normal, -velIntoSurface);
     }
     ```
   - Ground detection: `onGround = result.normal.y > 0.5;`
4. If no result: `onGround = false;`
5. Run collision check multiple times per tick (3-5 iterations) to handle corners and multi-surface contacts
6. Update capsule position again after collision resolution

G) **Camera wish direction:**
The existing code uses raw input for wish direction without considering camera yaw. Fix this:
```js
// Forward vector based on camera yaw (Y rotation)
const forward = new THREE.Vector3(0, 0, -1); // Three.js default forward is -Z
forward.applyAxisAngle(new THREE.Vector3(0, 1, 0), this.camera_yaw);
// Right vector perpendicular to forward
const right = new THREE.Vector3(1, 0, 0);
right.applyAxisAngle(new THREE.Vector3(0, 1, 0), this.camera_yaw);
// Wish direction = input.forward * forward + input.right * right
const wishDir = new THREE.Vector3();
wishDir.addScaledVector(forward, input.forward);
wishDir.addScaledVector(right, input.right);
wishDir.y = 0; // Keep wish direction horizontal
if (wishDir.lengthSq() > 0) wishDir.normalize();
```
